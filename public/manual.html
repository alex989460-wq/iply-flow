3 Passo a Passo da Instalação

Siga as etapas em ordem. Cada uma leva de 2 a 5 minutos.

1
Criar conta na Supabase

Acesse supabase.com e clique em Start your project. Você pode cadastrar-se com GitHub, Google ou email.

💡
DicaUse uma conta de email profissional que você acesse com frequência. Todos os avisos importantes chegam por email.
2
Criar novo projeto

No painel da Supabase, clique em New Project e preencha:

  • Name: super-gestor (ou o nome que preferir)
  • Database Password: gere uma senha forte e guarde em local seguro
  • Region: escolha a mais próxima dos seus usuários (South America - São Paulo)
  • Pricing Plan: Free para começar
⚠️
AtençãoA senha do banco NÃO pode ser recuperada. Anote-a em um gerenciador de senhas antes de continuar.
3
Aguardar criação do banco

A Supabase leva de 1 a 3 minutos para provisionar seu projeto. Uma tela de loading será exibida — não feche a aba.

✅
Pronto quando…Você for redirecionado para o dashboard do projeto com o menu lateral (Table Editor, SQL Editor, Auth, Storage, etc.).
4
Abrir o SQL Editor

No menu lateral esquerdo, clique no ícone SQL Editor (parece um >_). Em seguida clique em + New query para abrir uma nova aba de edição.

5
Executar o SQL completo

Vá até a seção 7 (SQL Completo) deste manual, clique em Copiar SQL, cole no SQL Editor da Supabase e clique em Run (ou Ctrl+Enter).

⚠️
Erros comunsSe aparecer "extension vector does not exist", vá em Database → Extensions e ative vector, pgcrypto, uuid-ossp e http manualmente antes de reexecutar.
💡
DicaSe o script for muito grande, execute em blocos: primeiro os enums e tabelas, depois constraints, funções, triggers e policies.
6
Configurar Storage

O SQL da etapa 5 já cria os 3 buckets automaticamente. Confirme em Storage → Buckets:

  • reseller-assets (público)
  • evolution-media (privado)
  • evolution-stickers (privado)

Detalhes na seção 7.

7
Copiar as chaves da Supabase

No menu Project Settings → API, copie:

  • Project URL — algo como https://xxxx.supabase.co
  • anon / public key — chave pública, segura para o frontend
  • service_role key — chave secreta, NUNCA exposta no frontend
🔒
SegurançaA service_role key ignora todas as políticas de RLS. Use somente em Edge Functions ou backend seguro.
8
Configurar variáveis de ambiente

Na raiz do projeto, crie um arquivo .env com o conteúdo:

.env
VITE_SUPABASE_URL="https://SEU-PROJETO.supabase.co"
VITE_SUPABASE_PUBLISHABLE_KEY="sua-anon-key-aqui"
VITE_SUPABASE_PROJECT_ID="SEU-PROJETO"

Veja todas as variáveis na seção 8.

9
Executar o projeto

No terminal, dentro da pasta do projeto:

terminal
# instalar dependências
npm install
# ou: bun install

# rodar em desenvolvimento
npm run dev
# ou: bun dev

# build de produção
npm run build

Abra http://localhost:8080 no navegador.

10
Verificar funcionamento

Cadastre-se na tela de login. O primeiro usuário criado vira automaticamente admin (trigger auto_grant_first_admin). Faça um tour rápido pelo Dashboard, Clientes e Configurações.

🎉
Parabéns!Se você chegou até aqui e o dashboard abriu, a instalação foi concluída com sucesso.
📦
Instalação nova x migraçãoSe você precisa entrar com emails antigos e ver clientes/revendedores já existentes, não basta executar o schema. Siga também a seção 6 (Migração e Acessos).

4 Configuração da Supabase

Onde encontrar cada chave e como usá-la corretamente.

🔗 Project URL

Em Project Settings → API → Project URL. Formato: https://[project-ref].supabase.co. Vai na variável VITE_SUPABASE_URL.

🔑 Anon Key (publishable)

Em Project Settings → API → Project API keys → anon public. É segura para uso no navegador — todas as regras de acesso são controladas por RLS. Vai na variável VITE_SUPABASE_PUBLISHABLE_KEY.

🛡️ Service Role Key

Em Project Settings → API → service_role (secret). Concede acesso total ignorando RLS. Guarde apenas em Edge Functions como secret (SUPABASE_SERVICE_ROLE_KEY). Nunca no frontend.

6 Migração e Acessos

Como recuperar clientes, planos, revendedores e login antigo depois de instalar em um novo backend.

⚠️
ImportanteO schema.sql cria a estrutura do sistema, mas não copia clientes, planos, servidores, arquivos, usuários de login, senhas, admins ou revendedores antigos. Esses dados precisam ser migrados separadamente.

🔐 Por que meu email admin não entra?

Os usuários de autenticação ficam em uma área protegida do backend. Ao criar um projeto novo, o email antigo não existe automaticamente e a senha antiga não é reaproveitada por SQL comum.

  • Crie/cadastre o admin com o mesmo email usado antes.
  • Depois execute access-repair.sql informando esse email.
  • O script cria/ajusta profiles, user_roles e reseller_access.

👥 Por que os revendedores não acessam?

Cada revendedor também precisa existir no Auth do novo projeto. O vínculo de acesso depende do novo auth.users.id, não apenas do email visível na tela.

  • Cadastre ou convide cada revendedor com o mesmo email antigo.
  • Preencha a lista de emails no access-repair.sql.
  • Execute o script e confira a validade em reseller_access.

🗄️ Por que meus clientes/dados não aparecem?

As regras de segurança mostram dados pelo dono. Se os dados foram importados com UUID antigo, as telas ficam vazias mesmo que as linhas existam no banco.

As colunas críticas são: user_id, created_by, owner_id, parent_reseller_id, used_by e approved_by.

💡
Regra práticaDepois de criar os usuários no novo projeto, mapeie email → novo auth.users.id e reatribua os clientes/configurações ao UUID novo do admin ou do revendedor correto.

conferência de usuários e acessos
SELECT
  u.email,
  p.full_name,
  array_agg(DISTINCT ur.role) FILTER (WHERE ur.role IS NOT NULL) AS roles,
  ra.is_active,
  ra.access_expires_at,
  ra.credits
FROM auth.users u
LEFT JOIN public.profiles p ON p.user_id = u.id
LEFT JOIN public.user_roles ur ON ur.user_id = u.id
LEFT JOIN public.reseller_access ra ON ra.user_id = u.id
GROUP BY u.email, p.full_name, ra.is_active, ra.access_expires_at, ra.credits
ORDER BY u.email;

CREATE OR REPLACE FUNCTION public.bulk_update_customers(usernames text[], server_ids uuid[], due_dates date[], statuses text[], screen_counts integer[], plan_ids uuid[])
 RETURNS integer
 LANGUAGE plpgsql
 SECURITY DEFINER
 SET search_path TO 'public'
AS $function$
DECLARE
  updated_count int := 0;
  i int;
BEGIN
  FOR i IN 1..array_length(usernames, 1) LOOP
    UPDATE customers SET
      server_id = server_ids[i],
      due_date = COALESCE(due_dates[i], due_date),
      status = statuses[i]::customer_status,
      screens = screen_counts[i],
      plan_id = plan_ids[i]
    WHERE username = usernames[i];
    IF FOUND THEN updated_count := updated_count + 1; END IF;
  END LOOP;
  RETURN updated_count;
END;
$function$

CREATE OR REPLACE FUNCTION public.get_dashboard_stats_optimized()
 RETURNS json
 LANGUAGE plpgsql
 SECURITY DEFINER
 SET search_path TO 'public'
AS $function$
DECLARE
  result JSON;
  sp_now TIMESTAMP := (NOW() AT TIME ZONE 'America/Sao_Paulo');
  today_date DATE := sp_now::date;
  tomorrow_date DATE := (sp_now + interval '1 day')::date;
  yesterday_date DATE := (sp_now - interval '1 day')::date;
  month_start DATE := date_trunc('month', sp_now)::date;
  current_user_id UUID := auth.uid();
BEGIN
  SELECT json_build_object(
    'totalCustomers', COALESCE(SUM(1), 0),
    'activeCustomers', COALESCE(SUM(CASE WHEN status = 'ativa' AND due_date >= today_date THEN 1 ELSE 0 END), 0),
    'inactiveCustomers', COALESCE(SUM(CASE WHEN status = 'inativa' THEN 1 ELSE 0 END), 0),
    'suspendedCustomers', COALESCE(SUM(CASE WHEN status = 'suspensa' THEN 1 ELSE 0 END), 0),
    'dueTodayCustomers', COALESCE(SUM(CASE WHEN due_date = today_date AND status IN ('ativa','inativa') THEN 1 ELSE 0 END), 0),
    'dueTomorrowCustomers', COALESCE(SUM(CASE WHEN due_date = tomorrow_date AND status IN ('ativa','inativa') THEN 1 ELSE 0 END), 0),
    'overdueOneDayCustomers', COALESCE(SUM(CASE WHEN due_date = yesterday_date AND status IN ('ativa','inativa') THEN 1 ELSE 0 END), 0),
    'overdueCustomers', COALESCE(SUM(CASE WHEN due_date < today_date AND status IN ('ativa','inativa') THEN 1 ELSE 0 END), 0),
    'newCustomersThisMonth', COALESCE(SUM(CASE WHEN created_at >= month_start THEN 1 ELSE 0 END), 0),
    'monthlyProjection', COALESCE(SUM(
      CASE WHEN status = 'ativa' AND due_date >= today_date THEN 
        COALESCE(custom_price, (SELECT price FROM plans WHERE plans.id = customers.plan_id), 0)
      ELSE 0 END
    ), 0)
  ) INTO result
  FROM customers
  WHERE created_by = current_user_id OR is_admin();
  
  RETURN result;
END;
$function$

CREATE OR REPLACE FUNCTION public.get_monthly_revenue()
 RETURNS json
 LANGUAGE plpgsql
 SECURITY DEFINER
 SET search_path TO 'public'
AS $function$
DECLARE
  result JSON;
  sp_now TIMESTAMP := (NOW() AT TIME ZONE 'America/Sao_Paulo');
  month_start DATE := date_trunc('month', sp_now)::date;
  today_date DATE := sp_now::date;
  current_user_id UUID := auth.uid();
BEGIN
  SELECT json_build_object(
    'monthlyRevenue', COALESCE(SUM(amount), 0),
    'todayRevenue', COALESCE(SUM(CASE WHEN payment_date = today_date THEN amount ELSE 0 END), 0),
    'todayPaymentCount', COALESCE(SUM(CASE WHEN payment_date = today_date THEN 1 ELSE 0 END), 0)
  ) INTO result
  FROM payments p
  WHERE payment_date >= month_start
    AND EXISTS (
      SELECT 1 FROM customers c 
      WHERE c.id = p.customer_id 
        AND (c.created_by = current_user_id OR is_admin())
    );
  
  RETURN result;
END;
$function$

CREATE OR REPLACE FUNCTION public.match_ai_knowledge_candidates(_user_id uuid, query_embedding vector, match_threshold double precision DEFAULT 0.86, match_count integer DEFAULT 3)
 RETURNS TABLE(id uuid, canonical_question text, similarity double precision)
 LANGUAGE sql
 STABLE SECURITY DEFINER
 SET search_path TO 'public'
AS $function$
  SELECT c.id, c.canonical_question, 1 - (c.embedding <=> query_embedding) AS similarity
  FROM public.ai_knowledge_candidates c
  WHERE c.user_id = _user_id
    AND c.embedding IS NOT NULL
    AND 1 - (c.embedding <=> query_embedding) >= match_threshold
  ORDER BY c.embedding <=> query_embedding
  LIMIT match_count;
$function$

CREATE OR REPLACE FUNCTION public.prevent_duplicate_customer_username()
 RETURNS trigger
 LANGUAGE plpgsql
 SECURITY DEFINER
 SET search_path TO 'public'
AS $function$
DECLARE
  normalized_username text;
  conflict_name text;
BEGIN
  normalized_username := public.normalize_customer_username(NEW.username);
  IF normalized_username IS NULL OR NEW.created_by IS NULL THEN
    RETURN NEW;
  END IF;

  SELECT c.name INTO conflict_name
  FROM public.customers c
  WHERE c.created_by = NEW.created_by
    AND public.normalize_customer_username(c.username) = normalized_username
    AND c.id <> COALESCE(NEW.id, '00000000-0000-0000-0000-000000000000'::uuid)
  LIMIT 1;

  IF conflict_name IS NOT NULL THEN
    RAISE EXCEPTION 'duplicate_customer_username:%', normalized_username
      USING ERRCODE = '23505', DETAIL = 'Já existe cliente com este usuário para este revendedor.';
  END IF;

  RETURN NEW;
END;
$function$

CREATE OR REPLACE FUNCTION public.prevent_duplicate_pending_customer_username()

CREATE OR REPLACE FUNCTION public.renew_customer_due_date()
 RETURNS trigger
 LANGUAGE plpgsql
 SECURITY DEFINER
 SET search_path TO 'public'
AS $function$
DECLARE
  plan_duration INTEGER;
  months_to_add INTEGER;
  current_due DATE;
  new_due DATE;
  original_day INTEGER;
  sp_today DATE := (now() AT TIME ZONE 'America/Sao_Paulo')::date;
  should_process BOOLEAN := false;
BEGIN
  IF NEW.confirmed = true THEN
    IF TG_OP = 'INSERT' THEN
      should_process := true;
    ELSIF TG_OP = 'UPDATE' AND OLD.confirmed = false THEN
      should_process := true;
    END IF;
  END IF;

  IF should_process THEN
    SELECT duration_days INTO plan_duration
    FROM public.plans p
    JOIN public.customers c ON c.plan_id = p.id
    WHERE c.id = NEW.customer_id;
    
    IF plan_duration IS NOT NULL THEN
      CASE plan_duration
        WHEN 30 THEN months_to_add := 1;
        WHEN 90 THEN months_to_add := 3;
        WHEN 180 THEN months_to_add := 6;
        WHEN 365 THEN months_to_add := 12;
        ELSE months_to_add := NULL;
      END CASE;

      SELECT due_date INTO current_due FROM public.customers WHERE id = NEW.customer_id;
      
      IF current_due IS NULL OR current_due < sp_today THEN
        current_due := sp_today;
      END IF;

      IF months_to_add IS NOT NULL THEN
        original_day := EXTRACT(DAY FROM current_due);
        new_due := current_due + (months_to_add || ' months')::interval;
        IF EXTRACT(DAY FROM new_due) <> original_day THEN
          new_due := (date_trunc('month', new_due))::date - 1;
        END IF;
      ELSE
        new_due := current_due + (plan_duration || ' days')::interval;
      END IF;

      UPDATE public.customers
      SET due_date = new_due, status = 'ativa'
      WHERE id = NEW.customer_id;
    END IF;
  END IF;
  RETURN NEW;
END;
$function$

11 Solução de Problemas

Erros mais comuns e como resolver.

ERROR: extension "vector" does not exist

Causa: a extensão pgvector não está ativada.

Solução: vá em Database → Extensions, procure por vector e clique em Enable. Reexecute o SQL.

permission denied for table X

Causa: faltam GRANTs no schema public.

Solução: o SQL fornecido já inclui os grants padrão via policies. Se persistir, execute:
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO authenticated;

Tela em branco após npm run dev

Causa: variáveis do .env ausentes ou incorretas.

Solução: abra o console (F12), procure por "SUPABASE_URL is undefined". Confirme que o .env existe na raiz do projeto e reinicie o dev server.

Cannot read properties of undefined (reading 'auth')

Causa: chave anon inválida ou URL errada.

Solução: confira novamente as chaves em Supabase → Project Settings → API. Copie exatamente sem espaços extras.

Row-level security policy violation

Causa: usuário tentando acessar dados sem role/policy adequada.

Solução: confirme que o primeiro usuário virou admin. Cheque a tabela user_roles — deve haver uma linha com role='admin'.

Login antigo não funciona ou revendedor não acessa

Causa: usuários de login e senhas não são migrados pelo schema.sql.

Solução: crie/cadastre os emails no Auth do novo projeto e execute manual/access-repair.sql para recriar perfil, role e acesso de revendedor.

Clientes existem no banco, mas não aparecem no sistema

Causa: dados importados ficaram vinculados ao UUID antigo do dono.

Solução: mapeie o email do revendedor para o novo auth.users.id e atualize colunas como created_by, user_id e owner_id. Veja a seção Migração e Acessos.

Edge Function não responde

Causa: secrets não configurados no painel.

Solução: em Project Settings → Edge Functions → Secrets, adicione as credenciais necessárias para a função (ver seção 8).

12 FAQ

Perguntas frequentes.

Posso rodar o sistema totalmente grátis?
Sim, o plano Free da Supabase é suficiente para começar. Você pagará apenas se ultrapassar os limites (500MB de banco, 1GB de storage, 500k invocações de Edge Function/mês).
Posso usar outro banco em vez da Supabase?
Tecnicamente sim, pois é PostgreSQL puro. Mas você perderá Auth, Storage, Realtime e Edge Functions — que exigiriam reescrita significativa. Não recomendado.
Como faço backup do banco?
A Supabase faz backups automáticos diários no plano Pro. No Free, use o botão de export ou implemente a Edge Function auto-backup já inclusa no projeto.
Como adiciono um segundo administrador?
Após o primeiro admin, use o SQL:
INSERT INTO public.user_roles (user_id, role) VALUES ('uuid-do-usuario', 'admin');
Por que meus clientes antigos não aparecem depois da instalação?
Porque o schema instala a estrutura, mas não importa dados. Além disso, clientes e configurações precisam estar vinculados ao novo UUID do dono nas colunas created_by, user_id ou owner_id.
Consigo reaproveitar as senhas antigas dos revendedores?
Não por SQL comum. Por segurança, os usuários devem ser criados/convidados novamente no Auth do novo projeto, usando o mesmo email e definindo nova senha ou fluxo de recuperação.
Preciso saber programar para usar?
Não para uso do sistema. Para instalar, basta seguir este manual passo a passo. Para customizar, é útil saber React + TypeScript.
Como atualizar para uma nova versão?
Git pull do repositório, npm install e execute apenas as novas migrations SQL fornecidas no changelog. Nunca reexecute o schema completo em produção.
Posso fazer deploy em produção onde?
Recomendamos Vercel, Netlify ou Cloudflare Pages para o frontend. O backend (Supabase) já é gerenciado.
Como faço quando algo dá errado que não está no manual?
Verifique os logs em Supabase → Logs → Edge Functions e no console do navegador (F12). A grande maioria dos erros mostra uma mensagem que aponta a causa.
